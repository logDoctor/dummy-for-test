# OpenTelemetry(OTel) 핵심 개념 및 아키텍처

OpenTelemetry(OTel)는 애플리케이션의 동작 상태를 파악하기 위한 **원격 분석 데이터(Telemetry)를 생성, 수집, 내보내는 표준화된 프레임워크**입니다. 

기존에는 모니터링 툴(Datadog, New Relic, Application Insights 등)마다 각기 다른 규칙과 코드를 작성해야 했지만, OpenTelemetry를 도입함으로써 **단일한 표준(세계 공통어)** 코드로 데이터를 추출하고 원하는 백엔드 모니터링 시스템으로 전송할 수 있게 되었습니다.

---

## 1. 텔레메트리(Telemetry)의 3대 요소
OpenTelemetry는 애플리케이션의 상태를 3가지 주요 데이터 유형(Signal)으로 정의하고 추출합니다.

### ① Traces (분산 추적)
> **"사용자의 요청이 어디서부터 어떻게 흘러갔는가?"**

*   **설명:** 사용자 요청 하나가 여러 마이크로서비스, 함수, 데이터베이스를 거쳐가는 흐름을 하나의 선으로 연결하여 보여주는 데이터입니다.
*   **작동 방식:**
    *   요청이 시작될 때 고유한 **`Trace ID`**가 발급됩니다.
    *   요청이 특정 작업 코루틴(A함수 -> B서버 -> C DB)을 거칠 때마다 **`Span`**이라는 작업 단위를 기록합니다.
    *   이 Span 안에는 "누가, 언제 시작/종료했는지, 상태 코드" 등의 속성(Attributes)이 기록됩니다. (우리가 앞서 미들웨어에 추가한 5W1H 정보가 바로 이 Span의 속성으로 들어갑니다.)

### ② Metrics (메트릭)
> **"시스템의 현재 상태나 누적된 수치는 어떠한가?"**

*   **설명:** CPU 사용량, 현재 접속 중인 유저 수, 1분당 결제 성공 횟수 등 숫자로 집계(Aggregation)되어 모니터링되는 측정값입니다.
*   **작동 방식:** 카운터(덧셈), 게이지(현재값), 히스토그램(분포도) 등의 방식으로 데이터를 차곡차곡 누적해 둡니다.

### ③ Logs (로그)
> **"그 순간 시스템 내부에서 어떤 일이 정확히 발생했는가?"**

*   **설명:** 개발자가 남기는 텍스트 형태의 메시지입니다. (예: `[ERROR] NullPointerException 발생`, `[INFO] 유저 정보 조회 완료`)
*   **작동 방식 및 OTel의 강력함 (Log Correlation):** OTel 환경에서는 단순 텍스트 로그에 그치지 않습니다. **로그가 발생한 시점의 현재 진행 중인 Trace ID와 Span ID가 로그에 자동으로 주입**됩니다. 이를 통해 특정 에러 로그를 클릭하면, 그 에러를 발생시킨 최초의 HTTP 요청(Trace) 흐름 전체를 한 번에 조회할 수 있습니다.

## 2. OTel의 작동 아키텍처
애플리케이션 안에서 일어난 이벤트가 모니터링 화면(예: Application Insights)까지 도달하는 과정은 다음의 3단계를 거칩니다.

### STEP 1: Instrumentation (계측 - "데이터 만들기")
*   **자동 계측 (Auto-Instrumentation):** 코드 수정 없이 애플리케이션에 찰싹 달라붙어 동작합니다. (예: Java Agent, Python의 `configure_azure_monitor()`) HTTP 요청, DB 쿼리를 감지해 자동으로 Span을 열고 닫는 역할을 합니다.
*   **수동 계측 (Manual):** 개발자가 코드 내에서 직접 `tracer.startActiveSpan("비즈니스_로직")`를 호출하여 원하는 특정 타이밍의 데이터를 커스텀하게 추출하는 방식입니다.

### STEP 2: SDK (모니터링 데이터 모으기)
*   애플리케이션 내부에서 만들어진 수치와 비즈니스 로그 조각들을 곧바로 보내지 않고, 내부 캐시나 메모리(임시 저장소)에 모아둡니다. (Batching) 성능 저하를 방지하기 위해 일정한 주기나 크기가 되면 방출(Export)을 준비합니다.

### STEP 3: Exporter (내보내기)
*   모아둔 데이터를 특정 포맷(OTLP, JSON 등)으로 변환해서 최종 도착지(Azure Application Insights, Datadog 등)로 전송하는 우체부 역할입니다.
*   우리가 예제 코드에서 사용한 Azure Monitor Distro 패키지 안에는 Azure에 최적화된 형식으로 바로 전송해 주는 전용 Exporter가 이미 장착되어 있습니다.

## 3. 핵심 아키텍처 강점: 문맥 전파 (Context Propagation)

OpenTelemetry의 가장 강력한 기능은 분산된 시스템 간의 **문맥 전파(Context Propagation)**입니다. 백엔드가 여러 언어로 구성된 마이크로서비스라고 가정해 봅시다.

1.  사용자의 요청이 클라이언트 앱에서 **Python 백엔드 서버**로 들어오면, OTel은 **`Trace ID: 1234`**를 생성합니다.
2.  Python 서버가 내부 로직을 처리하다가 데이터 조회를 위해 **Java 서버**를 HTTP 클라이언트로 호출합니다.
3.  이때, OTel은 HTTP 헤더(W3C Trace Context 규격) 속에 `Trace ID: 1234`를 몰래 담아 전송합니다.
4.  Java 서버에 설치된 OTel Agent가 HTTP 헤더를 읽고, *"아, 이건 내가 새로 시작한 요청이 아니라 `1234번`의 연장선(자식 흐름)이구나!"* 라고 인식합니다.
5.  결과적으로 모니터링 툴(Application Insights)에서는 **Python 서버의 작업 기록 옆에 자바 서버의 작업 기록이 선으로 하나로 예쁘게 이어져서(End-to-End Trace) 시각화**됩니다.

즉, **어떠한 프로그래밍 언어나 프레임워크를 사용하든 표준화된 모양으로 추출하고, 서로 다른 시스템끼리 단일한 요청 ID를 물고 다니며 추적망을 형성해 주는 것**이 OpenTelemetry 아키텍처의 존재 이유이자 핵심 원리입니다.
