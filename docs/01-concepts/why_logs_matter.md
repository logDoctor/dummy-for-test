# 왜 로그가 중요한가

## 목적
운영 관점에서 로그의 가치를 한 문서로 빠르게 이해하기 위한 요약입니다.

---

## 1. 장애 원인 추적

분산 환경에서는 요청 하나가 API Gateway → 인증 서버 → 비즈니스 서비스 → DB → 외부 결제 API를 거칩니다.
로그 없이는 이 중 어디서 터졌는지 알 수 없습니다.

**핵심:** Trace ID (요청 단위 식별자)가 있어야 재현이 어려운 문제를 역추적할 수 있습니다.

```
사용자가 "결제 실패" 신고
→ Trace ID로 전체 경로 조회
→ "외부 결제 API 타임아웃 503" 원인 발견 (3초 소요)
```

로그가 없었다면? 서버 전체를 하나씩 뒤져야 하는 수시간짜리 수동 작업이 됩니다.

---

## 2. 비즈니스 의사결정

구조화된 로그를 쓰면 개발자가 아닌 비즈니스 팀도 쿼리로 데이터를 볼 수 있습니다.

| 질문 | KQL로 답하는 방법 |
|---|---|
| "어느 화면에서 결제 이탈이 많나?" | `AppTraces`에서 `Audit_Action == "PAYMENT_ABANDON"` 집계 |
| "모바일 vs 웹 에러율 차이가 있나?" | `AppRequests`에서 `client_type`별 실패율 비교 |
| "오늘 새벽 배포 후 응답 시간이 늘었나?" | `AppRequests`에서 배포 시간 기준 `DurationMs` 추세 비교 |

**핵심:** "누가(Who) 어디서(Where) 무엇을(What) 했는지"를 지표화하면 제품 개선 사이클이 빨라집니다.

---

## 3. 보안/감사 대응

- **접근 이력:** 누가 언제 어떤 데이터를 조회/수정했는지
- **권한 변경:** 관리자 권한 부여/회수 기록
- **민감 데이터 접근:** 인사 정보, 계약서, 개인정보 열람 이력

이런 로그는 사고 발생 후 "범인을 찾는 유일한 증거"입니다.  
금융권(금감원), 의료(HIPAA), 공공(정보보안 지침) 등은 법적으로 1~5년 보존을 요구합니다.

---

## 좋은 로그의 최소 조건

| 항목 | 설명 |
|---|---|
| **구조화(JSON) 포맷** | 텍스트 덤프 대신 Key-Value로 저장해야 검색 가능 |
| **공통 필드 표준화** | `service`, `environment`, `user_id`, `trace_id`는 모든 로그에 공통 포함 |
| **수집/보관/삭제 정책 명시** | 무한 보관은 비용 폭탄 → 데이터 등급별로 기간 분리 |
| **검색 저장소 + 장기 보관 분리** | 분석용(LAW)과 아카이브(Blob Storage)를 용도에 맞게 설계 |
