# 현대 개발 환경에 필요한 로그, 어떻게 만들어야 할까?

과거에는 개발자가 `print("결제 완료")`처럼 단순 텍스트로 일기를 쓰듯 로그를 남겼습니다. 그러나 현대 개발 환경(분산 시스템, 클라우드, 마이크로서비스)에서는 이렇게 남긴 로그는 쓸모가 없습니다.

현대 사회가 요구하는 **'쓸모 있는 로그'를 만드는 3가지 규칙**은 다음과 같습니다.

---

## 1. 텍스트가 아닌 '구조화된 로그(Structured Logging)'로 남겨라

**❌ 과거의 방식 (단순 텍스트 로그)**
*   `[2024-03-10 15:00:01] 사용자 user123 님이 50000원을 결제했습니다. (IP: 192.168.0.1)`
*   **문제점:** "어제 5만원 이상 결제한 사람의 IP 목록 줘"라고 검색하려면 개발자가 정규표현식(Regex)을 짜서 텍스트를 파싱해야 합니다. 엄청난 낭비입니다.

**✅ 현대의 방식 (JSON 형태의 구조화된 로그)**
*   로그 메시지 본문은 짧게 남기되, 핵심 정보는 **Key-Value(사전)** 형태로 분리하여 주입합니다.
```json
{
  "Timestamp": "2024-03-10T15:00:01Z",
  "Message": "결제 완료",
  "LogLevel": "INFO",
  "CustomDimensions": {
    "UserID": "user123",
    "Amount": 50000,
    "ClientIP": "192.168.0.1"
  }
}
```
*   **장점:** Log Analytics Workspace 같은 중앙 데이터베이스에서 `| where CustomDimensions.Amount > 50000`처럼 엑셀(SQL) 검색하듯 0.1초 만에 깔끔하게 데이터를 뽑아낼 수 있습니다.

---

## 2. 5W1H (육하원칙) 컨텍스트를 설계해라

단순히 "에러 났음"이 아니라, 이 로그가 **누구에 의해, 어디서, 어떻게** 일어났는지 '문맥(Context)'을 반드시 포함해야 합니다.

*   **Who (누가):** 사용자 ID, 세션 토큰, 클라이언트 IP 주소 등 (*예: user_id: 8891*)
*   **Where (어디서):** 어느 서버, 어느 마이크로서비스, 어느 URI 엔드포인트인지 (*예: service_name: order-api, url: /api/checkout*)
*   **What/Why (무엇을/왜):** 장바구니에 아이템을 담았는지, 결제 실패 원인이 잔액 부족인지 등
*   **How (어떻게):** HTTP Method(GET, POST), 호출 방식(브라우저, 모바일 앱 버전)

### 💡 실무 꿀팁: "미들웨어(Middleware)로 한 번에 처리하기"
이 5W1H를 모든 비즈니스 함수마다 일일이 `logger.info(..., user="A", ip="127...")` 적는 것은 미친 짓입니다.
**앞서 우리가 파이썬, 자바 샘플 코드에서 했던 것처럼 라우터 앞단의 '미들웨어(글로벌 필터)'에서 요청이 들어올 때 한 번만 5W1H를 싹 다 긁어서 공통 컨텍스트(Context) 주머니에 넣어두는 것**이 핵심입니다. 그러면 그 요청 안에서 발생하는 모든 로그에는 이 정보가 자동으로 도장 찍히듯 붙어 나갑니다.

---

## 3. Correlation ID (상관관계 ID, 일명 Trace ID)를 전파해라

결제 API를 호출했는데, 내부적으로 재고 확인 API -> 결제 DB -> 이메일 발송 서버 순으로 4개의 시스템을 거친다고 가정해 봅시다. 
결제가 실패했을 때, 4개의 서버에 흩어진 에러 로그들을 어떻게 하나로 이어붙일 수 있을까요? 바로 **Trace ID**입니다.

1.  사용자가 처음 API를 찌를 때, 맨 앞단 서버가 **`Trace ID: 999xyz`** 라는 번호표를 만듭니다.
2.  이 서버가 다른 서버를 호출할 때 **HTTP 헤더**에 이 번호표를 몰래 넣어 보냅니다. (이것이 OpenTelemetry의 핵심입니다.)
3.  번호표를 넘겨받은 모든 서버는 자신이 남기는 로그 덩어리에 이 `Trace ID: 999xyz`를 박아 넣습니다.

*   **결과:** 에러가 났을 때 모니터링 툴(App Insights) 검색창에 `Trace ID: 999xyz`만 치면, 4대의 서버에서 발생한 모든 로그와 지연 시간 그래프가 **원인부터 결과까지 하나의 스토리라인으로 엮여서** 쫙 펼쳐집니다.

---

### 결론
현대 개발에서 로그를 만든다는 것은, "열심히 글씨(설명)를 적는 것"이 아닙니다. 
**"JSON 형태의 빈칸(Key)을 잘 설계하고, 5W1H 데이터를 미들웨어에서 자동 수집하게 만든 뒤, Trace ID로 모든 마이크로서비스를 실처럼 꿰어버리는 파이프라인(OpenTelemetry)을 구축하는 것"**입니다.
